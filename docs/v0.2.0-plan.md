# Atelier v0.2.0 — Tailwind v4 support

> **Goal**: Atelier classify / audit / atlas correctly handle Tailwind v4 projects (CSS-first config, `@theme` blocks, `oklch()` palette) without regressing v3 behavior.

Status: **planning** — not started. v0.1.0 shipped 2026-05-08 covers Tailwind v3 only.

## Scope

What changes in Tailwind v4 that Atelier needs to handle:

| v3 → v4 change | Affects |
|---|---|
| `tailwind.config.{js,ts}` → `@theme {...}` blocks in CSS | classify (registry), audit (token-source detection), atlas (default-DNA examples) |
| Default palette in `oklch()` (vs hex / rgb / hsl) | atlas (palette examples), audit (color contrast / format checks) |
| `--color-*` CSS variables exposed by default | classify (registry-strict mode needs to read CSS vars), audit |
| `@source` / `@variant` / `@utility` directives | minor — Atelier doesn't currently parse these |
| JS config deprecated (still opt-in via `@config`) | classify must NOT regress on legacy projects still using `tailwind.config.ts` |

What does NOT change:
- Class names — `bg-zinc-900` still means the same thing in v4. The "raw palette ref" detection in `@atelier-oss/classify` keeps working.
- DESIGN.md spec surface — DESIGN.md tokens are framework-agnostic. The spec doesn't change just because Tailwind did.
- `@google/design.md@0.1.1` upstream pin — track upstream for any v4-aware additions; otherwise Atelier extensions handle it.

## Phase ladder

Each phase ships one measurable thing. Gate clears before next starts.

### Phase 0 — Detection oracle (4-8 hours)

Build a v4-vs-v3 project detector. Single function, used by classify / audit / atlas / cli. **Gate**: 100% correct on a 12-repo fixture (6 v3, 6 v4).

```ts
detectTailwindVersion(repoPath: string): 'v3' | 'v4' | 'mixed' | 'none'
```

Heuristics, in order:
1. `@theme {...}` block in any `.css` / `.scss` file → v4
2. `tailwindcss@^4` in `package.json` deps → v4
3. `tailwind.config.{js,ts,cjs,mjs}` exists AND no `@theme` → v3
4. `tailwindcss@^3` in deps → v3
5. None of the above → `none`
6. v3 + v4 signals both present → `mixed` (warn loudly)

Fixture lives in `benchmarks/fixtures/tailwind-version/`.

### Phase 1 — Registry extension for `@theme` blocks (1-2 days)

`@atelier-oss/classify` v2-broad mode currently extracts the token registry from `tailwind.config.{js,ts}`, CSS variables, and DESIGN.md. Extend to parse `@theme {...}` block declarations.

**Gate**: classify run on the 6 v4 fixtures yields a registry that includes the `@theme`-defined tokens (parity oracle: every `@theme` declaration appears as a registry entry).

Implementation notes:
- `@theme` is a CSS at-rule; parse with a CSS AST (`postcss` already in deps via design.md? — verify) or a focused regex if scope is small enough.
- Declarations look like `--color-primary: oklch(0.6 0.15 240)` → registry key `primary`, value preserved verbatim.
- Multiple `@theme` blocks merge (last write wins per Tailwind v4 semantics).

### Phase 2 — Atlas oklch migration (4-8 hours)

`@atelier-oss/atlas` ships 7 build-category default-DNA palettes (currently in HSL or hex). Add an `oklch` field per category — keep HSL/hex for v3 projects.

**Gate**: every category in `packages/atlas/src/categories/*.ts` has `palette.v3` (existing) and `palette.v4` (oklch). atlas-fingerprint output contains both; consumer picks based on detected Tailwind version.

### Phase 3 — Audit additions (1 day)

`@atelier-oss/audit` token-source detection: add a section that reports presence/absence of `@theme` blocks in v4 projects and warns when a v4 project still has a `tailwind.config.ts` (mixed mode).

**Gate**: audit run on the 6 v4 fixtures emits the expected `tailwind_v4_theme_block_present` / `mixed_mode_detected` findings.

### Phase 4 — Spec addendum (2-4 hours)

Append a § to `spec/DESIGN.md.spec.md` documenting:
- `@theme` blocks count as a registry source for the precedence rule
- `oklch()` is an accepted color format
- Detection oracle behavior

No new finding codes. Existing `ATELIER_PRECEDENCE_VIOLATION` and `ATELIER_MISSING_ROLE` continue to apply.

### Phase 5 — CLI + docs (4 hours)

- `atelier classify --tailwind=v4|v3|auto` (default `auto` uses Phase 0 detector).
- README "Quickstart" gets a v4 note.
- CHANGELOG entry via changesets.

**Gate**: `pnpm -r build` clean, `pnpm test` green, `python3 -m benchmarks.parity_check` still 60/60 on the v3 corpus (no regression).

### Phase 6 — Benchmark v3 (1 day)

Add a small v4 corpus (6-10 repos) to `benchmarks/`. Re-run the three-arm primary gate on v4 projects only.

**Gate**: directional confirmation — DESIGN.md arm still beats shadcn-default arm by ≥10pp on v4 projects (looser than v0.1.0's +15pp because n is smaller). If gate fails, ship v0.2.0 anyway with a documented caveat — v4 detection is the user-visible win, not a re-validated lift number.

### Phase 7 — Release v0.2.0

`pnpm changeset` for each touched package, merge to main, tag `v0.2.0`, CI publishes via the existing direct `pnpm -r publish` flow.

## Non-goals for v0.2.0

- Lightning CSS / OXC parser swaps. Stay on existing CSS parsing path.
- Custom `@variant` / `@utility` rule analysis. Out of scope until users ask.
- Generative Phase 2 study (causation vs correlation). Separate track, not gating.

## Estimated total

~5-7 working days end-to-end. Phase 1 (registry extension) and Phase 6 (benchmark) are the two unknowns.

- Tailwind v4 is GA as of late 2025; `@theme` semantics are stable.
