# v2 result analysis (2026-05-07)

Pre-registered in `benchmarks/spec-v2.md`. Three classifier modes ran
against the locked 24-repo corpus:

| Mode | DESIGN.md | shadcn-default | raw-palette | Primary Δ | Primary verdict | Secondary | Secondary verdict |
|---|---|---|---|---|---|---|---|
| v1 | 80.0% | 65.0% | 60.7% | +14.95pp | FAIL | +31.75% | FAIL |
| v2-strict | 72.5% | **0.0%** | **0.0%** | +72.52pp | PASS (degenerate) | n/a | n/a |
| **v2-broad** | 74.0% | 60.6% | **29.8%** | **+13.34pp** | **FAIL** | **+148.03%** | **PASS** |

## v2-strict is degenerate — disregard its PASS

v2-strict defines the registry as "DESIGN.md `colors:` keys only." By
construction, no repo without a DESIGN.md can have a non-empty registry.
shadcn-default and raw-palette repos have no DESIGN.md, so their registries
are uniformly empty, every class is demoted to RAW, and conformance is 0%
across both arms. The +72.52pp delta is a measurement artifact, not a real
effect. **v2-strict tells us nothing.**

The pre-registration listed strict and broad as a sensitivity pair. It
turns out they're not symmetric checks — strict is structurally degenerate
because it asks a tautological question. Future spec versions should
either drop strict or redefine it as "any *named* token registry" so the
two modes test different breadth assumptions, not different existence
assumptions.

## v2-broad is the honest answer

v2-broad uses the union of DESIGN.md keys, tailwind.config color keys,
and CSS custom property names. This catches every reasonable way a
project might declare "these are my tokens."

- **DESIGN.md (74.0%)** dropped slightly from v1 (80.0%) because v1 was
  crediting some classes that don't appear in the project's registry —
  in-flight typos, abandoned tokens, and classes referencing removed
  variables. The 6pp drop is the *fair correction*.
- **shadcn-default (60.6%)** held roughly steady (v1 = 65.0%). Shadcn
  projects already have well-populated tailwind.config + CSS vars
  registries. Modest tightening, not the source of the v1→v2 narrative.
- **raw-palette (29.8%)** collapsed from v1 = 60.7% — almost half the
  arm's "token" credits in v1 were leakage. Repos like base-camp-tampa
  (96.8% v1 → 2.2% v2-broad) used custom Tailwind classes with first
  segments that *looked* semantic but were never declared anywhere.
  The leakage hypothesis is confirmed.

## Gate verdicts under v2-broad

- **Primary (DESIGN.md vs shadcn-default, ≥+15pp absolute)**: FAIL by 1.66pp. DESIGN.md adds ~13pp over the shadcn baseline. Real, modest, sub-threshold.
- **Secondary (DESIGN.md vs raw-palette, ≥+40% relative)**: PASS at +148%. DESIGN.md (or any explicit token contract) is dramatically better than no contract.

## What this means

1. **DESIGN.md's value is real but smaller than the MVB suggested.** The MVB's +105% relative lift compared "with DESIGN.md" against a mixed control. v2-broad's +148% relative lift compares "with DESIGN.md" against repos with literally no token contract. Both are large, but the second is a cleaner test of the underlying claim.
2. **The shadcn convention does most of the work.** Most projects in the wild use shadcn defaults + tailwind.config extensions, which already provide a token contract. DESIGN.md adds ~13pp on top. That's the calibrated number.
3. **The pre-registered +15pp absolute gate was set high relative to the actual effect.** This is a gate-design issue, not a hypothesis-rejection. The hypothesis has signal; the gate was tighter than the signal.

## Implications for the upstream PR

If opened at v2-broad numbers, the PR should:

- Lead with the +148% relative lift over raw-palette (the dramatic, defensible number).
- Acknowledge the +13pp absolute gap vs shadcn-default — note that DESIGN.md's incremental value over shadcn is modest, but additive.
- Frame DESIGN.md as a *complement* to existing shadcn conventions, not a replacement.
- Cite spec-v2 pre-registration to head off "you tightened the classifier post-hoc" objections — both v1 and v2 results are reported, and v2 was committed before running.

This story is more nuanced than "DESIGN.md is a 105% revolution," but it's defensible and it's the story the data tells.

## Next move

The pre-registered primary gate (v2-broad, ≥+15pp absolute) FAILED. The
plan's documented action on FAIL is "block public launch, write the
analysis." Both done. From here, three real forks remain — but they're
strategic, not technical:

1. **Recalibrate the gate.** A +13pp gap on a +15pp pre-reg is close enough that ratifying the result and shipping the upstream PR with the calibrated numbers is defensible. The pre-reg respected the strict reading; recalibration would be a soft override.
2. **Run a Phase 2 generative study.** Replace the observational design with a causal one: same corpus, AI generates code with and without DESIGN.md in context, measure generation-time conformance. This is the test the upstream PR audience would actually want and cleanly answers "does DESIGN.md cause better generation?" Cost: ~3-5 days, ~$50-100 in API.
3. **Stand down.** The hypothesis has signal but doesn't clear the gate; the project can stay private and the upstream PR can wait until a cleaner study is possible.

Each fork is a strategic call.
